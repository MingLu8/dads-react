The golden rule of React, up front:

A React app is just functions that return HTML, and re-run when their data changes. That's 80% of it. The other 20% is how you store the data (state) and when the function re-runs.

Day 1 — Components, JSX, Props & Rendering Lists

Goal: understand what a component is, read JSX fluently, and render the message list from data.

1.1 A component is a function that returns markup

Open src/components/Chat.tsx. Strip away the details and it's just:

export function Chat() {
  return (
    <>
      <main className="dads__chat"> ... </main>
      <footer className="dads__composer"> ... </footer>
    </>
  );
}

That's a component: a function whose name is Capitalized and that returns JSX (HTML-looking syntax). You use it like a tag: <Chat />. (The <>...</> is a Fragment — a wrapper that renders no extra DOM node, for when you need to return siblings without a div.) See it used in the route table, AppRoutes.tsx: <Route path="/" element={<Chat />} />.

🅰️→⚛️ Angular component = a @Component class + a separate .html template + a CSS file. React component = one function that returns its own markup. No decorator, no separate template file.

1.2 JSX is HTML with JavaScript holes

JSX looks like HTML but it's really JavaScript. The differences that trip people up:

HTML / AngularJSX (React)Why
class="dads"className="dads"class is a reserved JS word
{{ title }}{title}single braces = "drop a JS value here"
(click)="submit()"onClick={submit}camelCase, pass the function
[disabled]="busy"disabled={busy}{} = a JS expression
*ngIf / @if{cond && <x/>}just JavaScript
*ngFor / @forarray.map(...)just JavaScript

Anything inside { } is live JavaScript. In Chat.tsx:

<div className="bubble__text">{m.content}</div>   {/* {m.content} = a JS value */}

1.3 Rendering a list — .map() instead of *ngFor

This is the single most common React pattern. In Chat.tsx:

{messages.map((m, i) => (
  <div key={i} className={`bubble ${m.role === 'user' ? 'bubble--user' : 'bubble--dads'}`}>
    <div className="bubble__text">{m.content}</div>
  </div>
))}

messages is an array; .map() turns each item into a <div>. React needs a key on each item (a stable id) so it can update the list efficiently — the equivalent of Angular's trackBy.

🅰️→⚛️ *ngFor="let m of messages" → {messages.map(m => <.../>)}. [class.bubble--user]="m.role==='user'" → a template string: className={`bubble ${cond ? 'bubble--user' : 'bubble--dads'}`}.

1.4 Conditional rendering — && and ? :

No *ngIf directive — you use plain JS expressions that evaluate to JSX (or nothing). Straight from Chat.tsx:

{messages.length === 0 && (
  <div className="dads__empty"><p>Ask me something!</p></div>
)}

{busy && (
  <div className="bubble bubble--dads"><div className="bubble__text typing">Dads is thinking...</div></div>
)}

cond && <jsx/> renders the JSX only when cond is true (because false renders nothing). For either/or, use a ternary: {loggedIn ? <Chat/> : <Login/>} — exactly what AuthedApp.tsx does.

1.5 Props — passing data into a component

A component receives inputs called props (like Angular @Input()s). A parent passes them as attributes; the child reads them as a function argument:

// parent:
<AuthProvider value={{ getToken, userEmail, onLogout }}>...</AuthProvider>

// child (AuthProvider.tsx):
export function AuthProvider({ value, children }: { value: AuthContextValue; children: React.ReactNode }) { ... }

Two props worth naming early: children (whatever you nest between the tags) and callback props like onLogout — functions a child calls to talk back to its parent. In React, "passing data down" and "passing callbacks up" both happen through props.

🅰️→⚛️ @Input() userEmail: string → a field in the props object. @Output() logout = new EventEmitter() → a prop that is a function (onLogout). <ng-content> → the children prop.

But notice what our app does not do: it doesn't thread getToken/messages down through five layers of props. Past a couple of levels, prop-drilling gets painful — so app-wide data (auth, the conversation) lives in Context instead (Day 3). Chat takes no props at all; it pulls what it needs from context. Keep props for local parent→child data; reach for context when "everyone needs it."

💡 Deep dive: key — and why index keys can bite

React needs a key on each list item to track which DOM node maps to which piece of data across re-renders. The key must be stable and unique per item. We use the array index (key={i}) in Chat, which is fine here because messages are only ever appended — item 0 stays item 0 forever.

But index keys break the moment a list can reorder, filter, or insert at the top. Delete the first item and every remaining index shifts by one, so React thinks every row changed identity — it can reuse the wrong DOM node, and any per-row state (a half-typed input, an open menu) jumps to the wrong row. Rule of thumb: use a real id (key={msg.id}) whenever items can move; index keys only for static, append-only lists. Never use Math.random() as a key — it changes every render, defeating the point.

💡 Deep dive: components must be pure during render

Your component function can run many times, whenever React decides — so the render body must be a pure calculation of UI from props/state. No side effects in the body:

function Chat() {
  fetch('/api/...')   // ❌ runs on every render — a storm of requests
  setDraft('x')       // ❌ setState during render → infinite re-render loop
  return <div/>;
}

Side effects belong in event handlers (onClick) or in useEffect (Day 3). The mental split: render = "describe the UI"; handlers/effects = "do things." This purity is exactly what lets React re-render freely and fast — and it's what the React Compiler (this app uses it) relies on to auto-optimize your components.

Day 2 — State, Hooks, Events & Forms (the heart of React)

Goal: make the UI interactive. Yesterday we rendered data; today the data changes and the screen updates itself. This is where React clicks.

2.1 The mental model: state in, UI out

When state changes, React re-runs your component function and updates the screen to match.

You never touch the DOM directly (no document.getElementById, no *ngModel two-way magic). You change a variable; React re-renders. The tool for "a variable that triggers re-render" is the useState hook.

2.2 useState — a value React watches

In Chat.tsx:

const [draft, setDraft] = useState('');

Read that as: "give me a state variable draft (starts as '') and a setter setDraft." When you call setDraft('hello'), React re-renders Chat and draft is now 'hello'. Always change state via the setter — never draft = 'hello' (React wouldn't know).

useChat.ts holds the conversation state:

const [messages, setMessages] = useState<ChatMessage[]>([]);  // the conversation
const [busy, setBusy]         = useState(false);              // is a request in flight?

🅰️→⚛️ Angular re-renders automatically when a class field changes (zone.js / signals). React needs you to call the setter so it knows. useState ≈ an Angular signal(): count() → count, count.set(5) → setCount(5).

Rule of hooks: call hooks at the top level of a component (or another hook), never inside loops, conditions, or nested functions. (That's why AuthedApp/PlainApp are split — so useAuth0() is only called when the provider exists. More in the Day 2 deep dive.)

2.3 Events — onClick, onChange, onKeyDown

Events are props you pass a function to. From Chat.tsx:

<button onClick={submit}>Send</button>
<button onClick={reset}>New Chat</button>

Pass the function (onClick={submit}), don't call it (onClick={submit()} would run it during render — a classic bug). Need arguments? Wrap in an arrow: onClick={() => open(a.id)} (see HistoryPage.tsx).

2.4 Controlled inputs — React's version of [(ngModel)]

This is the one concept Angular devs must rewire. In Angular, [(ngModel)]="draft" binds both ways automatically. In React you wire the two directions explicitly — and that's considered a feature (the input's value is always exactly your state). From Chat.tsx:

<textarea
  value={draft}                            // state → input  (display)
  onChange={e => setDraft(e.target.value)} // input → state  (capture typing)
  onKeyDown={onKeyDown}
  disabled={busy}
/>

This is a controlled component: draft is the single source of truth. Want to clear the box after sending? Just setDraft('') — the textarea follows. See submit():

const submit = () => {
  const content = draft.trim();
  if (!content || busy) return;
  setDraft('');        // clears the textarea
  void send(content);  // fire the request (Day 3)
};

🅰️→⚛️ [(ngModel)]="draft" = value={draft} + onChange={e => setDraft(e.target.value)}. Two-way binding is just two one-way bindings you can see.

2.5 useRef — a value that does not trigger re-render

Sometimes you need to remember something that isn't UI. In useChat.ts:

const sessionId = useRef<string | null>('');
// ...later, inside send():
if (!sessionId.current) sessionId.current = await createSession(token);

sessionId.current survives re-renders but changing it does not re-render (unlike state). Perfect for "plumbing" like the backend session id — the user doesn't need to see it, but every message must reuse it so the server keeps our conversation together. Think of it as a private instance field on the old Angular service.

2.6 Custom hooks — your "service" layer

A custom hook is just a function named useSomething that uses other hooks. useChat.ts bundles all conversation logic so the component stays about display:

export function useChat() {
  const [messages, setMessages] = useState<ChatMessage[]>([]);
  const [busy, setBusy]         = useState(false);
  const sessionId = useRef<string | null>('');
  const { getToken } = useAuth();                 // pull the token from auth context

  const send = useEffectEvent(async (content: string) => {
    /* append the user msg, flip busy on, create session if needed,
       call the API, append the reply, flip busy off */
  });
  const reset  = useCallback(() => { setMessages([]); sessionId.current = null; }, []);
  const resume = useEffectEvent(async (id: string) => { /* load a past session's history */ });

  return { messages, busy, send, reset, resume };
}

The component just consumes it: const { messages, busy, send, reset } = useChatContext(); (we go through a shared context rather than calling useChat() directly — that's the Day 3 Context lesson, and there's a real bug we hit by getting it wrong).

🅰️→⚛️ This is the React answer to an Angular service. Angular: @Injectable class + DI into the constructor. React: a useXxx function you call at the top of a component (or share via Context). Same goal (reusable logic separated from the view), different mechanism.

React 19 note — useEffectEvent & the React Compiler. You'll notice send/resume use useEffectEvent (a stable callback that always "sees" the latest props/state without needing a dependency array). And you'll see very few useCallback/useMemo calls — this app enables the React Compiler (babel-plugin-react-compiler), which auto-memoizes for you. While learning, don't sweat manual memoization; focus on state and effects.

💡 Deep dive: why the Rules of Hooks exist (the most important 3 minutes)

React identifies hooks by call ORDER, not by name. It stores each component's hook data in an ordered list — picture numbered slots. Every re-render it walks the list in the same order and hands slot 0 to the first hook call, slot 1 to the second. The names are yours; React just counts.

So hide a hook behind a condition and the slots shift:

// ❌ illegal
if (loggedIn) { const [draft] = useState(''); } // present some renders, gone others
const [busy] = useState(false);                  // slot 0 OR slot 1 depending on loggedIn!

Now busy sometimes reads the slot that holds draft. You get the runtime error "Rendered fewer hooks than expected." Stable call order is the entire contract.

Legal ways to be conditional: put the if inside the hook (useEffect(() => { if (x) … }, [x])), or call all hooks first and return early after them (what AuthedApp does), or split into two components. This is why our app splits AuthedApp / PlainApp: useAuth0() can't be called conditionally, and only works inside <Auth0Provider> (which main.tsx mounts only when auth is enabled). So we choose the whole component at the top. The eslint-plugin-react-hooks rule flags violations as you type — trust it.

💡 Deep dive: state is a snapshot (updates are async + batched)

This trips up everyone. setBusy(true) does not change busy on the next line — it schedules a re-render. During the current render your variables are a frozen snapshot:

const [count, setCount] = useState(0);
function onClick() {
  setCount(count + 1);   // count is 0 → schedules 1
  setCount(count + 1);   // count is STILL 0 → schedules 1 (not 2!)
  console.log(count);    // logs 0 — the change isn't visible until the next render
}

Both calls read the same snapshot, so you get 1, not 2. The fix is the functional updater — pass a function that receives the latest value:

setCount(prev => prev + 1);
setCount(prev => prev + 1);   // now → 2

That's exactly why useChat appends with setMessages(prev => [...prev, msg]) rather than setMessages([...messages, msg]): between the user message and the assistant reply there are several updates, and the functional form always sees the freshest array. React also batches the setters fired in one event into a single re-render.

💡 Deep dive: never mutate state — replace it

React decides whether to re-render by comparing the reference (identity) of your state, not its contents. Mutating in place is therefore invisible:

messages.push(newMsg);                    // ❌ same array reference → React sees "no change"
setMessages(messages);                    // ❌ still the same reference → no re-render
setMessages(prev => [...prev, newMsg]);   // ✅ NEW array → reference changed → re-render

Same for objects: setUser(u => ({ ...u, name: 'X' })), never user.name = 'X'. Treat state as read-only and always produce a new value — that's the deep reason the spread ... is everywhere.

Day 3 — Talking to the Backend, Effects, Context & Shipping

Goal: connect the UI to the real API, understand useEffect and Context, see how the session and auth are wired, and build for production.

3.1 Fetching data — fetch + async/await

No HttpClient, no RxJS Observable — React uses the browser's fetch. The whole API layer is src/api.ts. Two calls power a chat turn:

export async function createSession(token?: string): Promise<string> {
  const response = await fetch('/api/session', { method: 'POST', headers: header(token), body: '{}' });
  if (!response.ok) throw new Error(`Session request failed with status ${response.status}`);
  const data = await response.json() as { sessionId: string };
  return data.sessionId;
}

export async function sendChat(sessionId: string, message: string, token?: string): Promise<ChatApiResponse> {
  const chatMessage = { content: message, role: 'user' };
  const response = await fetch('/api/chat', {
    method: 'POST',
    headers: header(token),                         // { 'Content-Type': 'application/json', Authorization? }
    body: JSON.stringify({ chatMessage, sessionId }),
  });
  if (!response.ok) throw new Error(`Chat request failed with status ${response.status}`);
  return await response.json() as ChatApiResponse;  // { message: string }
}

useChat.send orchestrates it: optimistically add the user's message, flip busy on, create the session if we don't have one yet, await the reply, append it, flip busy off (in finally), and on error append a friendly message. Notice there's no manual subscribe/unsubscribe — await reads like normal code.

🅰️→⚛️ Angular this.http.post(url, body).subscribe(...) (an Observable you must unsubscribe). React: const res = await fetch(...) then await res.json(). For one-shot calls, promises are simpler than Observables. The /api proxy (vite.config.ts → https://localhost:7293) replaces Angular's proxy.conf.json.

Why a server session id? The frontend sends only the new message plus the sessionId; the .NET backend stores the running history and replays it to the model. So reusing the same sessionId across messages is what gives the assistant memory within a conversation — keep that in mind for Day 3's context deep dive.

3.2 useEffect — running code as a side effect of rendering

The chat is event-driven (you click Send), so it doesn't need useEffect. But the History page does: it loads your past conversations once when it opens. HistoryPage.tsx:

useEffect(() => {
  let active = true;
  getSessions()
    .then(a => { if (active) setSessions(a); })
    .catch(() => { if (active) setError(true); });
  return () => { active = false; };   // cleanup: ignore a late response after unmount
}, []);   // [] = "run once on mount"

The dependency array is the key:

  • [] → run once (on mount). ≈ Angular ngOnInit.
  • [x] → run on mount and whenever x changes.
  • (omitted) → run after every render (rarely what you want).
  • return a function → cleanup. ≈ Angular ngOnDestroy / unsubscribing.

🅰️→⚛️ ngOnInit → useEffect(fn, []). ngOnChanges → useEffect(fn, [theInput]). ngOnDestroy → the cleanup function you return from the effect.

3.3 Context — dependency injection, React-style

Two things in this app need to be shared by many components without prop-drilling: auth (who you are, how to get a token) and the conversation (so the History page can resume into the Chat page). React's tool for that is Context: a Provider at the top, consumers anywhere below.

Auth context. main.tsx wraps the app in Auth0's provider (when enabled), and AuthedApp.tsx adapts it into our own small AuthProvider so the rest of the app reads a tidy { getToken, userEmail, onLogout }:

// AuthedApp.tsx — translate Auth0 into our context value
<AuthProvider value={{
  getToken: async () => (await getIdTokenClaims())?.__raw,
  userEmail: user?.email,
  onLogout: () => logout({ logoutParams: { returnTo: window.location.origin } }),
}}>
  <AppRoutes />
</AuthProvider>

Any component then calls the useAuth() hook — Layout shows the email and Logout button; useChat grabs getToken to attach the bearer token. No prop-drilling.

Chat context. ChatContext.tsx creates one useChat() instance and shares it:

export function ChatProvider({ children }: { children: ReactNode }) {
  const chat = useChat();                       // ← the single source of truth
  return <ChatContext.Provider value={chat}>{children}</ChatContext.Provider>;
}
export function useChatContext() {
  const ctx = useContext(ChatContext);
  if (!ctx) throw new Error('useChatContext must be used with <ChatProvider>');
  return ctx;
}

AppRoutes wraps the routes in <ChatProvider>, so Chat and HistoryPage share the same messages and the same sessionId. That's what lets History's resume(id) load a conversation that the Chat page then displays.

🅰️→⚛️ This is React's dependency injection. Angular: providedIn: 'root' + inject into a constructor (one shared instance). React: wrap a <Provider> and call its hook (useAuth, useChatContext). The Provider is what makes it a singleton — see the deep dive.

💡 Deep dive: one Provider = one instance (a real bug we hit)

Calling a custom hook does not share state — every call to useChat() creates its own independent state. The Provider exists precisely so everyone reads the same instance via useChatContext().

We learned this the hard way. Chat.tsx originally called useChat() directly instead of useChatContext() — spinning up a second conversation with its own sessionId ref, separate from the provider's. The symptoms:

  • The assistant "forgot" everything after navigation. Chat is a routed component; leaving / for /history and back unmounts and remounts it, resetting its private sessionId to empty — so the next message started a brand-new server session with no history.
  • Resume did nothing. HistoryPage called resume() on the shared instance, but Chat rendered from its own — so the resumed conversation never showed.

The fix is one line in Chat.tsx:

// ❌ before — a private, throwaway instance
const { messages, busy, send, reset } = useChat();
// ✅ after — the shared instance from <ChatProvider>
const { messages, busy, send, reset } = useChatContext();

Because <ChatProvider> wraps <Routes> (and so never unmounts during navigation), the shared sessionId now survives page changes. Lesson: a hook is a factory, not a singleton — if state must be shared, create it once in a Provider and consume it through the context hook everywhere else.

3.4 The auth on/off toggle (and the "rules of hooks" payoff)

auth.config.ts reads enabled from an env var. main.tsx chooses the tree:

authConfig.enabled
  ? <Auth0Provider ...><AuthedApp /></Auth0Provider>   // login gate, tokens on requests
  : <PlainApp />                                        // no auth (backend dev bypass)

Why two components (AuthedApp vs PlainApp)? Because hooks can't be conditional — you can't call useAuth0() only "sometimes." So we decide which component renders at the top, and each one calls its hooks unconditionally. PlainApp just renders <AppRoutes/>; AuthedApp gates on login first, then renders the same <AppRoutes/> wrapped in our AuthProvider.

💡 Deep dive: the dependency array & stale closures

The useEffect trap that catches everyone: if you read a prop or state inside the effect but forget to list it in deps, the effect keeps using the OLD value. Every render creates fresh copies of your functions that capture that render's values; an effect with [] captures only the first render's values, forever.

useEffect(() => {
  const id = setInterval(() => console.log(count), 1000);
  return () => clearInterval(id);
}, []);     // ❌ logs 0 forever — captured the first render's count
            // ✅ use [count] so it re-subscribes with the latest value

Rule: list every reactive value the effect uses. The exhaustive-deps lint rule tells you exactly what's missing — don't silence it, fix it. (React 19's useEffectEvent, which useChat uses, is the escape hatch for "I want the latest value but don't want to re-run on every change.")

💡 Deep dive: why your effect runs twice in dev (StrictMode)

main.tsx wraps the app in <StrictMode>. In development only, StrictMode deliberately mounts each component twice (mount → unmount → mount) to flush out bugs: effects that don't clean up, or that aren't safe to run twice. If you see an effect fire twice locally — a duplicate fetch, a double log — that's not a bug, it's StrictMode stress-testing you. The fix is never "remove StrictMode"; it's "return a cleanup function and make the effect idempotent" (exactly the active flag in HistoryPage). In production it runs once.

Day 3½ — Testing your app (Vitest + React Testing Library) 🧪

The testing tooling is already wired up — vitest.config.ts, src/setupTests.ts, and the test/test:run scripts in package.json — but the suite itself is yours to write. This unit shows how, using our real code. Writing these tests is the fastest way to lock in everything from Days 1–3.

The stack:

  • Vitest — the test runner (Vite-native, fast, Jest-compatible API).
  • React Testing Library (RTL) — render components and query them like a user would.
  • jsdom — a fake browser DOM so tests run in Node (no real browser needed).

🅰️→⚛️ Angular ships Karma + Jasmine + TestBed. React's common stack is Vitest (or Jest) + RTL. TestBed.createComponent → RTL's render. Jasmine spyOn → Vitest vi.fn() / vi.mock().

The golden rule of RTL:

Test behavior, not implementation. Find things the way a user does (by text, label, role) and assert on what they'd see — not on internal state or function names. This keeps tests from breaking when you refactor.

The testing pyramid — test at 3 levels

1. Pure functions — easiest. Test the api.ts fetch wrappers by stubbing the global fetch:

// src/api.test.ts
import { sendChat } from './api';

it('sendChat POSTs { chatMessage, sessionId } and returns the reply', async () => {
  const fetchMock = vi.fn().mockResolvedValue({ ok: true, json: async () => ({ message: 'hi' }) });
  vi.stubGlobal('fetch', fetchMock);

  const res = await sendChat('abc', 'hello');

  expect(res.message).toBe('hi');
  const [url, opts] = fetchMock.mock.calls[0];
  expect(url).toBe('/api/chat');
  expect(JSON.parse(opts.body)).toEqual({ chatMessage: { content: 'hello', role: 'user' }, sessionId: 'abc' });
});

vi.fn() makes a spy you can inspect (.mock.calls); vi.stubGlobal swaps out fetch so no real network happens.

2. Hooks — test logic without a UI. Use renderHook + act, and mock the whole api module so it never hits the network:

// src/hooks/useChat.test.ts
import * as api from '../api';
vi.mock('../api');                       // auto-replaces createSession/sendChat with spies
const mockedApi = vi.mocked(api);

it('send() appends the user message then the bot reply', async () => {
  mockedApi.createSession.mockResolvedValue('session-1');
  mockedApi.sendChat.mockResolvedValue({ message: 'Hello from Dads' });

  const { result } = renderHook(() => useChat());
  await act(async () => { await result.current.send('hi'); });

  expect(result.current.messages).toEqual([
    { role: 'user', content: 'hi' },
    { role: 'bot',  content: 'Hello from Dads' },
  ]);
});

act(...) tells React "I'm about to cause state updates — apply them before I assert." (Heads-up: useChat reads useAuth(), so render it inside a tiny <AuthProvider value={{}}> wrapper or mock ../hooks/useAuth.)

3. Components — the most valuable. Render the real component and drive it like a user with userEvent. Because Chat reads useChatContext(), wrap it in <ChatProvider> (and an <AuthProvider>):

// src/components/Chat.test.tsx
render(
  <AuthProvider value={{}}><ChatProvider><Chat /></ChatProvider></AuthProvider>
);

const user = userEvent.setup();
await user.type(screen.getByPlaceholderText(/how can i help/i), 'hello');
await user.click(screen.getByRole('button', { name: /send/i }));

expect(await screen.findByText('hello')).toBeInTheDocument();

Day 4 — Routing with React Router 🧭

Real apps have more than one screen. The app has three routes — Chat (/), History (/history), and Sources (/sources) — with nav between them. React doesn't ship a router; the standard is react-router-dom (already installed).

The route table — src/AppRoutes.tsx

<BrowserRouter>
  <ChatProvider>                                       {/* shared chat state for all pages */}
    <Routes>
      <Route element={<Layout />}>                       {/* layout route: shared shell */}
        <Route path="/"        element={<Chat />} />        {/* "/"         */}
        <Route path="/history" element={<HistoryPage />} /> {/* "/history"  */}
        <Route path="/sources" element={<Sources />} />     {/* "/sources"  */}
        <Route path="*"        element={<NotFound />} />     {/* anything else */}
      </Route>
    </Routes>
  </ChatProvider>
</BrowserRouter>
  • BrowserRouter — turns on history-API routing (clean URLs, no #). Wrap it once, near the top.
  • Routes / Route — match the URL to an element; path="*" is the catch-all.
  • Layout route — a <Route> that wraps children renders <Layout> once; each child renders into the Layout's <Outlet/>.
  • ChatProvider sits above Routes — that's deliberate: it never unmounts as you navigate, so the conversation and sessionId persist across pages (recall the Day 3 deep dive).

Navigating — NavLink, Link, useNavigate

src/components/Layout.tsx uses NavLink, which knows when it's active:

<NavLink to="/" end className={linkClass}>Chat</NavLink>
<NavLink to="/history" className={linkClass}>History</NavLink>
<NavLink to="/sources" className={linkClass}>Sources</NavLink>
<Outlet />   {/* the matched page renders here */}

And HistoryPage.tsx navigates from code after resuming a session:

const navigate = useNavigate();
const open = async (id: string) => {
  await chat.resume(id);   // load that conversation into the shared chat state
  navigate('/');           // jump to the Chat page to show it
};
  • Link / NavLink do client-side navigation — no full page reload. Use them instead of <a href>.
  • NavLink adds active styling automatically (end = exact match, needed for "/").
  • Route params: <Route path="ticket/:id" …> then const { id } = useParams();

🅰️→⚛️ RouterModule.forRoot(routes) → <BrowserRouter> + <Routes>. <router-outlet> → <Outlet/>. routerLink → <Link to>. routerLinkActive → NavLink's isActive. Router.navigate() → useNavigate(). ActivatedRoute.params → useParams().

💡 Deep dive: client-side routing needs a server fallback

BrowserRouter changes the URL without asking the server — it just swaps components in place. Great UX, but: if a user hard-refreshes /history, the browser asks your server for /history, which isn't a real file → 404. The fix is an SPA fallback: configure the host to serve index.html for all paths. Vite's dev server and vite preview do this automatically; on Nginx / Azure Static Web Apps you add a catch-all rewrite to index.html. Forget this and routing "works in dev, 404s in prod" — a classic.

⚠️ The 8 gotchas that bite every beginner (bookmark this)

These are the deep dives above, distilled. 90% of "why is React doing that?!" moments are one of these:

  1. Hooks must run in the same order every render — never call them in if/loops/nested fns. React matches hooks by call order, not name. (Day 2)
  2. State is a snapshot — setX(v) doesn't change x until the next render. Need the latest value? setX(prev => …).
  3. Never mutate state — build a new array/object ([...prev, x]); React compares by reference, so in-place edits don't re-render.
  4. List every effect dependency — a missing dep = stale value (a "stale closure"). Trust exhaustive-deps.
  5. Stable keys — real ids for lists that can reorder/filter; array index only for append-only lists.
  6. No side effects in the render body — put them in event handlers or useEffect; render must be pure.
  7. Effects run twice in dev (StrictMode) — make them idempotent and return a cleanup function; it's not a bug.
  8. A hook is a factory, not a singleton — calling useChat() twice makes two states. Share via a Provider and read it with useChatContext(). (This is the exact bug that made the assistant "forget" on navigation — Day 3 deep dive.)
