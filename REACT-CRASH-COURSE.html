Learn React in 3 Days — using the Dads chat app 🚀

Welcome! You already have a working app — the Dads internal AI chat assistant. We built it in Angular, then rebuilt the exact same app in React (this folder, dads-react). That's your superpower for learning: you're not learning React on a toy to-do list, you're learning it on a real app you already understand.

Who this is for: developers who know a little JavaScript/TypeScript (and maybe Angular) and want to be productive in React fast.

How it works: 3 core days, plus bonus units on testing (Day 3½) and routing (Day 4), and a hands-on build-it-yourself appendix. ~2–3 focused hours each. Every concept is shown in our app, with a short 🅰️ Angular → ⚛️ React translation and a 🛠️ Try it challenge. Do the challenges — that's where it sticks. (Prefer building over reading? Jump to the Appendix and construct the app step by step, then come back for the "why.")

The golden rule of React, up front:

A React app is just functions that return HTML, and re-run when their data changes. That's 80% of it. The other 20% is how you store the data (state) and when the function re-runs.

Day 1 — Components, JSX, Props & Rendering Lists

Goal: understand what a component is, read JSX fluently, and render the message list from data.

1.1 A component is a function that returns markup

Open src/components/Chat.tsx. Strip away the details and it's just:

export function Chat() {
  return (
    <div className="dads">
      <header className="dads__header"> ... </header>
      <main className="dads__chat"> ... </main>
      <footer className="dads__composer"> ... </footer>
    </div>
  );
}

That's a component: a function whose name is Capitalized and that returns JSX (HTML-looking syntax). You use it like a tag: <Chat />. See it used in PlainApp.tsx:

export function PlainApp() {
  return <Chat />;
}

🅰️→⚛️ Angular component = a @Component class + a separate .html template + a CSS file. React component = one function that returns its own markup. No decorator, no separate template file.

1.2 JSX is HTML with JavaScript holes

JSX looks like HTML but it's really JavaScript. The differences that trip people up:

HTML / Angular JSX (React) Why
class="dads" className="dads" class is a reserved JS word
{{ title }} {title} single braces = "drop a JS value here"
(click)="submit()" onClick={submit} camelCase, pass the function
[disabled]="busy" disabled={busy} {} = a JS expression
*ngIf / @if {cond && <x/>} just JavaScript
*ngFor / @for array.map(...) just JavaScript

Anything inside { } is live JavaScript. In Chat.tsx:

<p>Dadsniti internal assistant · React</p>     {/* static text */}
<div className="bubble__text">{msg.content}</div>  {/* {msg.content} = a JS value */}

1.3 Rendering a list — .map() instead of *ngFor

This is the single most common React pattern. In Chat.tsx:

{messages.map((msg, i) => (
  <div key={i} className={`bubble ${msg.role === 'user' ? 'bubble--user' : 'bubble--dads'}`}>
    <div className="bubble__text">{msg.content}</div>
  </div>
))}

messages is an array; .map() turns each item into a <div>. React needs a key on each item (a stable id) so it can update the list efficiently — the equivalent of Angular's trackBy.

🅰️→⚛️ *ngFor="let msg of messages" → {messages.map(msg => <.../>)}. [class.bubble--user]="msg.role==='user'" → className={\bubble ${cond ? 'bubble--user' : ''}`}`.

1.4 Conditional rendering — && and ? :

No *ngIf directive — you use plain JS expressions that evaluate to JSX (or nothing):

{messages.length === 0 && (
  <div className="dads__empty"><p>Ask me something…</p></div>
)}

{busy && (
  <div className="bubble bubble--dads"><div className="typing">Dads is thinking…</div></div>
)}

cond && <jsx/> renders the JSX only when cond is true (because false renders nothing). For either/or, use a ternary: {loggedIn ? <Chat/> : <Login/>} — exactly what AuthedApp.tsx does.

1.5 Props — passing data into a component

A component receives inputs called props (like Angular @Input()s). Look at the top of Chat.tsx:

interface ChatProps {
  getToken?: () => Promise<string | undefined>;
  userEmail?: string;
  onLogout?: () => void;
}

export function Chat({ getToken, userEmail, onLogout }: ChatProps) { ... }

The parent decides what to pass. PlainApp passes nothing (<Chat />); AuthedApp passes all three:

<Chat getToken={() => getAccessTokenSilently()} userEmail={user?.email} onLogout={() => logout(...)} />

Inside Chat, props are read-only. Notice {userEmail && <span>{userEmail}</span>} — the email chip only shows when a parent provided one.

🅰️→⚛️ @Input() userEmail: string → a field in the props object. @Output() logout = new EventEmitter() → a prop that is a function (onLogout). In React, "passing data down" and "passing callbacks up" both happen through props.

💡 Deep dive: key — and why index keys can bite

React needs a key on each list item to track which DOM node maps to which piece of data across re-renders. The key must be stable and unique per item. We use the array index (key={i}) in Chat, which is fine here because messages are only ever appended — item 0 stays item 0 forever.

But index keys break the moment a list can reorder, filter, or insert at the top. Delete the first item and every remaining index shifts by one, so React thinks every row changed identity — it can reuse the wrong DOM node, and any per-row component state (a half-typed input, an open menu) jumps to the wrong row. Rule of thumb: use a real id (key={msg.id}) whenever items can move; index keys only for static, append-only lists. Never use Math.random() as a key — it changes every render, defeating the whole point.

💡 Deep dive: components must be pure during render

Your component function can run many times, whenever React decides — so the render body must be a pure calculation of UI from props/state. No side effects in the body:

function Chat() {
  fetch('/api/...')   // ❌ runs on every render — a storm of requests
  setDraft('x')       // ❌ setState during render → infinite re-render loop
  return <div/>;
}

Side effects belong in event handlers (onClick) or in useEffect (Day 3). The mental split: render = "describe the UI"; handlers/effects = "do things." This purity is exactly what lets React re-render freely and fast.

Day 2 — State, Hooks, Events & Forms (the heart of React)

Goal: make the UI interactive. Yesterday we rendered data; today the data changes and the screen updates itself. This is where React clicks.

2.1 The mental model: state in, UI out

When state changes, React re-runs your component function and updates the screen to match.

You never touch the DOM directly (no document.getElementById, no *ngModel two-way magic). You change a variable; React re-renders. The tool for "a variable that triggers re-render" is the useState hook.

2.2 useState — a value React watches

In Chat.tsx:

const [draft, setDraft] = useState('');

Read that as: "give me a state variable draft (starts as '') and a setter setDraft." When you call setDraft('hello'), React re-renders Chat and draft is now 'hello'. Always change state via the setter — never draft = 'hello' (React wouldn't know).

useChat.ts has two more:

const [messages, setMessages] = useState<ChatMessage[]>([]);  // the conversation
const [busy, setBusy]         = useState(false);              // is a request in flight?

🅰️→⚛️ Angular re-renders automatically when a class field changes (zone.js / signals). React needs you to call the setter so it knows. useState ≈ an Angular signal(): count() → count, count.set(5) → setCount(5).

Rule of hooks: call hooks at the top level of a component (or another hook), never inside loops, conditions, or nested functions. (That's why AuthedApp/PlainApp are split — so useAuth0() is only called when the provider exists.)

2.3 Events — onClick, onChange, onKeyDown

Events are props you pass a function to. From Chat.tsx:

<button onClick={submit}>Send</button>
<button onClick={reset}>New chat</button>

Pass the function (onClick={submit}), don't call it (onClick={submit()} would run it during render — a classic bug). Need arguments? Wrap in an arrow: onClick={() => logout(...)}.

2.4 Controlled inputs — React's version of [(ngModel)]

This is the one concept Angular devs must rewire. In Angular, [(ngModel)]="draft" binds both ways automatically. In React you wire the two directions explicitly — and that's considered a feature (the input's value is always exactly your state):

<textarea
  value={draft}                          // state → input  (display)
  onChange={e => setDraft(e.target.value)} // input → state  (capture typing)
  onKeyDown={onKeyDown}
  disabled={busy}
/>

This is a controlled component: draft is the single source of truth. Want to clear the box after sending? Just setDraft('') — the textarea follows. See submit():

const submit = () => {
  const text = draft.trim();
  if (!text || busy) return;
  setDraft('');        // clears the textarea
  void send(text);     // fire the request
};

🅰️→⚛️ [(ngModel)]="draft" = value={draft} + onChange={e => setDraft(e.target.value)}. Two-way binding is just two one-way bindings you can see.

2.5 useRef — a value that does not trigger re-render

Sometimes you need to remember something that isn't UI. In useChat.ts:

const sessionId = useRef<string | null>(null);
// ...later:
if (!sessionId.current) sessionId.current = await createSession();

sessionId.current survives re-renders but changing it does not re-render (unlike state). Perfect for "plumbing" like the session id. Think of it as a private instance field on the old Angular service.

2.6 Custom hooks — your "service" layer

A custom hook is just a function named useSomething that uses other hooks. useChat.ts bundles all conversation logic so the component stays about display:

export function useChat(getToken?) {
  const [messages, setMessages] = useState<ChatMessage[]>([]);
  const [busy, setBusy] = useState(false);
  const sessionId = useRef<string | null>(null);

  const send = useCallback(async (text) => { /* ...append, call API, append reply... */ }, [busy, getToken]);
  const reset = useCallback(() => { setMessages([]); sessionId.current = null; }, []);

  return { messages, busy, send, reset };
}

The component just consumes it:

const { messages, busy, send, reset } = useChat(getToken);

🅰️→⚛️ This is the React answer to an Angular service. Angular: @Injectable class + DI into the constructor. React: a useXxx function you call at the top of a component. Same goal (reusable logic, separated from the view), different mechanism. useCallback just memoizes a function so it isn't recreated every render — a performance nicety, optional while learning.

💡 Deep dive: why the Rules of Hooks exist (the most important 3 minutes)

You met the rule in 2.2. Here's the mechanism — once you get it, you'll never write a hook bug again.

React identifies hooks by call ORDER, not by name. It stores each component's hook data in an ordered list — picture numbered slots. When Chat renders:

const { messages, busy } = useChat(getToken); // slot 0
const [draft, setDraft]  = useState('');      // slot 1

React records "slot 0 = useChat's state, slot 1 = draft." Every re-render it walks the list in the same order and hands slot 0 to the first hook call, slot 1 to the second. The names are yours; React just counts: 1st hook, 2nd hook, 3rd…

So hide a hook behind a condition and the slots shift:

// ❌ illegal
if (loggedIn) { const [draft] = useState(''); } // present some renders, gone others
const [busy] = useState(false);                  // slot 0 OR slot 1 depending on loggedIn!

Now busy sometimes reads the slot that holds draft. State leaks into the wrong hook and you get the runtime error "Rendered fewer hooks than expected." Stable call order is the entire contract.

Legal ways to be conditional: put the if inside the hook (useEffect(() => { if (x) … }, [x])), or call all hooks first and return early after them (what AuthedApp does), or split into two components.

This is why our app splits AuthedApp / PlainApp. useAuth0() (a) can't be called conditionally per the rule above, and (b) only works inside <Auth0Provider>, which main.tsx mounts only when auth is enabled. So we choose the whole component at the top: AuthedApp always calls useAuth0() (and is only ever rendered inside the provider); PlainApp never calls it. No hook is ever conditional — we pushed the branch up and out of the components. (Same reasoning as the Angular optional-inject we used, different mechanism.) The eslint-plugin-react-hooks rule flags violations as you type — trust it.

💡 Deep dive: state is a snapshot (updates are async + batched)

This trips up everyone. setBusy(true) does not change busy on the next line — it schedules a re-render. During the current render your variables are a frozen snapshot:

const [count, setCount] = useState(0);
function onClick() {
  setCount(count + 1);   // count is 0 → schedules 1
  setCount(count + 1);   // count is STILL 0 → schedules 1 (not 2!)
  console.log(count);    // logs 0 — the change isn't visible until the next render
}

Both calls read the same snapshot, so you get 1, not 2. The fix is the functional updater — pass a function that receives the latest value:

setCount(prev => prev + 1);
setCount(prev => prev + 1);   // now → 2

That's exactly why useChat appends with setMessages(prev => [...prev, msg]) rather than setMessages([...messages, msg]): between the user message and the assistant reply there are several updates, and the functional form always sees the freshest array. React also batches the setters fired in one event into a single re-render — another reason state can't change mid-function.

💡 Deep dive: never mutate state — replace it

React decides whether to re-render by comparing the reference (identity) of your state, not its contents. Mutating in place is therefore invisible:

messages.push(newMsg);                    // ❌ same array reference → React sees "no change"
setMessages(messages);                    // ❌ still the same reference → no re-render
setMessages(prev => [...prev, newMsg]);   // ✅ NEW array → reference changed → re-render

Same for objects: setUser(u => ({ ...u, name: 'X' })), never user.name = 'X'. Treat state as read-only and always produce a new value — that's the deep reason the spread ... is everywhere in React code.

Day 3 — Talking to the Backend, Effects, Auth & Shipping

Goal: connect the UI to the real API, understand useEffect and Context, see how auth is wired, and build for production.

3.1 Fetching data — fetch + async/await

No HttpClient, no RxJS Observable — React uses the browser's fetch. The whole API layer is src/api.ts:

export async function sendChat(sessionId, message, token?) {
  const res = await fetch('/api/chat', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json', ...(token && { Authorization: `Bearer ${token}` }) },
    body: JSON.stringify({ message, sessionId }),
  });
  if (!res.ok) throw new Error(`chat failed: ${res.status}`);
  return res.json();
}

useChat.send orchestrates it: optimistically add the user's message, flip busy on, await the reply, add it, flip busy off (in finally), and on error append a friendly message. Notice there's no manual subscribe/unsubscribe — await reads like normal code.

🅰️→⚛️ Angular this.http.post(url, body).subscribe(...) (an Observable you must unsubscribe). React: const res = await fetch(...) then await res.json(). For one-shot calls, promises are simpler than Observables. The /api proxy (vite.config.ts) replaces Angular's proxy.conf.json.

3.2 useEffect — running code as a side effect of rendering

Our chat is event-driven (you click Send), so it doesn't need useEffect. But it's a core hook, so here's the model: useEffect runs after render, and re-runs when its dependencies change. Use it to sync with the outside world (fetch on load, subscriptions, timers, document.title).

// Example: load sources status once when the component mounts.
useEffect(() => {
  fetch('/api/sources').then(r => r.json()).then(setSources);
}, []);   // [] = "run once on mount"

The dependency array is the key:

  • [] → run once (on mount). ≈ Angular ngOnInit.
  • [x] → run on mount and whenever x changes.
  • (omitted) → run after every render (rarely what you want).
  • return a function → cleanup. ≈ Angular ngOnDestroy / unsubscribing.

🅰️→⚛️ ngOnInit → useEffect(fn, []). ngOnChanges → useEffect(fn, [theInput]). ngOnDestroy → the cleanup function you return from the effect.

3.3 Context — dependency injection, React-style

Auth0 needs to share "is the user logged in?" with any component. React's tool for app-wide shared data is Context (a Provider at the top, consumers anywhere below). See main.tsx:

<Auth0Provider domain={...} clientId={...} authorizationParams={{ redirect_uri: window.location.origin }}>
  <AuthedApp />
</Auth0Provider>

Then any component inside can read it with the useAuth0() hook — no prop-drilling. AuthedApp.tsx:

const { isAuthenticated, loginWithRedirect, logout, user, getAccessTokenSilently } = useAuth0();
if (!isAuthenticated) return <LoginScreen onClick={() => loginWithRedirect()} />;
return <Chat getToken={() => getAccessTokenSilently()} userEmail={user?.email} onLogout={...} />;

🅰️→⚛️ This is React's dependency injection. Angular: providedIn: 'root' + inject into a constructor. React: wrap a <Provider> and call its hook (useAuth0, or useContext(MyContext)).

3.4 The auth on/off toggle (and the "rules of hooks" payoff)

auth.config.ts has enabled. main.tsx chooses the tree:

authConfig.enabled
  ? <Auth0Provider ...><AuthedApp /></Auth0Provider>   // login gate, tokens on requests
  : <PlainApp />                                        // no auth (backend dev bypass)

Why two components (AuthedApp vs PlainApp)? Because hooks can't be conditional — you can't call useAuth0() only "sometimes." So we decide which component renders at the top, and each one calls its hooks unconditionally. This is the same reasoning the Angular app used (optional-inject vs not).

💡 Deep dive: the dependency array & stale closures

The useEffect trap that catches everyone: if you read a prop or state inside the effect but forget to list it in deps, the effect keeps using the OLD value. This is a "stale closure." Every render creates fresh copies of your functions that capture that render's values; an effect with [] captures only the first render's values, forever.

useEffect(() => {
  const id = setInterval(() => console.log(count), 1000);
  return () => clearInterval(id);
}, []);     // ❌ logs 0 forever — captured the first render's count
            // ✅ use [count] so it re-subscribes with the latest value

Rule: list every reactive value the effect uses. The eslint-plugin-react-hooks exhaustive-deps rule tells you exactly what's missing — don't silence it, fix it. (And recall Day 2's functional setState(prev => …): it lets you update state without depending on it, sidestepping some stale-closure cases entirely.)

💡 Deep dive: why your effect runs twice in dev (StrictMode)

main.tsx wraps the app in <React.StrictMode>. In development only, StrictMode deliberately mounts each component twice (mount → unmount → mount) to flush out bugs: effects that don't clean up, or that aren't safe to run twice. If you see an effect fire twice locally — a duplicate fetch, a double log — that's not a bug, it's StrictMode stress-testing you. The fix is never "remove StrictMode"; it's "return a cleanup function and make the effect idempotent." In production it runs once.

Day 3½ — Testing your app (Vitest + React Testing Library) 🧪

You shipped a working app — now let's make sure it stays working. The app already has a full test suite (npm run test:run → 18 passing tests). Let's read it; it's the best way to learn testing.

The stack:

  • Vitest — the test runner (Vite-native, fast, Jest-compatible API). Config: vitest.config.ts.
  • React Testing Library (RTL) — render components and query them like a user would.
  • jsdom — a fake browser DOM so tests run in Node (no real browser needed).

🅰️→⚛️ Angular ships Karma + Jasmine + TestBed. React's common stack is Vitest (or Jest) + RTL. TestBed.createComponent → RTL's render. Jasmine spyOn → Vitest vi.fn() / vi.mock().

The golden rule of RTL:

Test behavior, not implementation. Find things the way a user does (by text, label, role) and assert on what they'd see — not on internal state or function names. This keeps tests from breaking when you refactor.

The testing pyramid — we test at 3 levels

1. Pure functions — easiest. src/api.test.ts tests the fetch wrappers by stubbing the global fetch:

it('sendChat POSTs message + sessionId and returns the reply', async () => {
  const fetchMock = vi.fn().mockResolvedValue({ ok: true, json: async () => ({ reply: 'hi', sessionId: 'abc' }) });
  vi.stubGlobal('fetch', fetchMock);

  const res = await sendChat('abc', 'hello');

  expect(res.reply).toBe('hi');
  const [url, opts] = fetchMock.mock.calls[0];
  expect(url).toBe('/api/chat');
  expect(JSON.parse(opts.body)).toEqual({ message: 'hello', sessionId: 'abc' });
});

vi.fn() makes a spy you can inspect (.mock.calls); vi.stubGlobal swaps out fetch so no real network happens.

2. Hooks — test logic without a UI. src/hooks/useChat.test.ts uses renderHook + act, and mocks the whole api module so it never hits the network:

vi.mock('../api');                       // auto-replaces createSession/sendChat with spies
const mockedApi = vi.mocked(api);

it('send() appends the user message then the assistant reply', async () => {
  mockedApi.createSession.mockResolvedValue('session-1');
  mockedApi.sendChat.mockResolvedValue({ reply: 'Hello from Dads', sessionId: 'session-1' });

  const { result } = renderHook(() => useChat());
  await act(async () => { await result.current.send('hi'); });

  expect(result.current.messages).toEqual([
    { role: 'user', content: 'hi' },
    { role: 'assistant', content: 'Hello from Dads', sources: undefined },
  ]);
});

act(...) tells React "I'm about to cause state updates — apply them before I assert." Wrap anything that triggers a re-render in it.

3. Components — the most valuable. src/components/Chat.test.tsx renders the real component and drives it like a user with userEvent:

it('sends a message and renders the user turn and the reply', async () => {
  const user = userEvent.setup();
  render(<Chat />);

  await user.type(screen.getByPlaceholderText(/how can i help/i), 'how many accounts?');
  await user.click(screen.getByRole('button', { name: /send/i }));

  expect(await screen.findByText('how many accounts?')).toBeInTheDocument();
  expect(await screen.findByText(/11,180 accounts/)).toBeInTheDocument();
});

Day 4 — Routing with React Router 🧭

Real apps have more than one screen. The app now has two routes — Chat (/) and Sources (/sources) — with nav between them. React doesn't ship a router; the standard is react-router-dom.

npm install react-router-dom

The route table — src/AppRoutes.tsx

<BrowserRouter>
  <Routes>
    <Route element={<Layout … />}>                      {/* layout route: shared shell */}
      <Route index element={<Chat … />} />              {/* "/"        */}
      <Route path="sources" element={<SourcesPage />} /> {/* "/sources" */}
      <Route path="*" element={<NotFound />} />          {/* anything else */}
    </Route>
  </Routes>
</BrowserRouter>
  • BrowserRouter — turns on history-API routing (clean URLs, no #). Wrap it once, near the top.
  • Routes / Route — match the URL to an element. index is the default child (/); path="*" is the catch-all.
  • Layout route — a <Route> that wraps children renders <Layout> once; each child renders into the Layout's <Outlet/>.

Navigating — NavLink, Link, useNavigate

src/components/Layout.tsx uses NavLink, which knows when it's active:

<NavLink to="/" end className={({ isActive }) => (isActive ? 'active' : '')}>Chat</NavLink>
<NavLink to="/sources">Sources</NavLink>
<Outlet />   {/* the matched page renders here */}
  • Link / NavLink do client-side navigation — no full page reload. Use them instead of <a href>.
  • NavLink adds active styling automatically (end = exact match, needed for "/").
  • Navigate from code after an action: const navigate = useNavigate(); navigate('/sources');
  • Route params: <Route path="ticket/:id" …> then const { id } = useParams();

How routing composes with auth

The auth gate stays outside the router: AuthedApp does the login check, then renders <AppRoutes/>; PlainApp renders <AppRoutes/> directly. "Are you logged in?" is decided once; routing handles "which page" after that.

🅰️→⚛️ RouterModule.forRoot(routes) → <BrowserRouter> + <Routes>. <router-outlet> → <Outlet/>. routerLink → <Link to>. routerLinkActive → NavLink's isActive. Router.navigate() → useNavigate(). ActivatedRoute.params → useParams().

💡 Deep dive: client-side routing needs a server fallback

BrowserRouter changes the URL without asking the server — it just swaps components in place. Great UX, but: if a user hard-refreshes /sources, the browser asks your server for /sources, which isn't a real file → 404. The fix is an SPA fallback: configure the host to serve index.html for all paths. Vite's dev server and vite preview do this automatically; on Nginx / Azure Static Web Apps you add a catch-all rewrite to index.html. Forget this and routing "works in dev, 404s in prod" — a classic.

⚠️ The 7 gotchas that bite every beginner (bookmark this)

These are the deep dives above, distilled. 90% of "why is React doing that?!" moments are one of these:

  1. Hooks must run in the same order every render — never call them in if/loops/nested fns. React matches hooks by call order, not name. (Day 2 deep dive)
  2. State is a snapshot — setX(v) doesn't change x until the next render. Need the latest value? setX(prev => …).
  3. Never mutate state — build a new array/object ([...prev, x]); React compares by reference, so in-place edits don't re-render.
  4. List every effect dependency — a missing dep = stale value (a "stale closure"). Trust exhaustive-deps.
  5. Stable keys — real ids for lists that can reorder/filter; array index only for append-only lists.
  6. No side effects in the render body — put them in event handlers or useEffect; render must be pure.
  7. Effects run twice in dev (StrictMode) — make them idempotent and return a cleanup function; it's not a bug.
